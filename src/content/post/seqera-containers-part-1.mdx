## The proposed change

In September 2024, nf-core proposed changing how modules received their default containers. Developers would edit one Conda `environment.yml`. nf-core automation would ask Wave to build the image, store it in Seqera Containers, and pin its address.

Pipeline users would still select Docker, Singularity, or Conda as before. Most of the change would affect module development and nf-core automation.

BioContainers and Bioconda made nf-core's existing system possible. We remain grateful to their contributors. Anaconda, Quay.io, and Galaxy also provided community hosting. AWS and Seqera supported the proposed service.

<Admonition type="tip" title="What nf-core proposed in 2024">

- Developers edit the Conda `environment.yml` instead of maintaining a separate container declaration.
- Wave builds Docker and Singularity images on demand.
- Seqera Containers stores the images behind stable addresses.
- nf-core records build inputs, logs, and Conda lock files.
- Renovate proposes package updates.
- Pipeline commands remain the same for users.

</Admonition>

This post explains the reason for the proposal. [Part 2](/posts/seqera-containers-part-2/) describes the planned implementation.

## How the existing system worked

Most nf-core modules included an `environment.yml` with packages from [Bioconda](https://bioconda.github.io/). BioContainers built images from those packages and published them through Quay.io and Galaxy.

A module developer found the matching Docker and Singularity addresses, then copied both into `main.nf`. A pipeline user pulled the selected image when Nextflow ran the process.

That system served nf-core well. Four parts became harder as the module collection grew.

### Multi-package images

Many processes need more than one package. A tool may call [samtools](https://www.htslib.org/) or pipe output through [pigz](https://zlib.net/pigz/).

BioContainers handled these combinations through mulled images. A developer added package hashes to a shared file, waited for review and CI, found the new image address, and updated the module. [Luke Pembleton documented the process](https://lpembleton.rbind.io/posts/mulled-biocontainers/), and [Moriz E. Beber built a helper](https://midnighter.github.io/mulled) for finding image names.

The process worked, but each package update required several manual steps. It often took longer than the module change itself.

### Waiting for Singularity images

BioContainers published Docker images to Quay.io soon after a build. A nightly Galaxy job produced the matching Singularity image. Module testing could pause until that second image appeared.

The proposal moved both formats into one on-demand build. A developer could request them from the same environment file.

### Development API failures

nf-core tooling queried the BioContainers API to find image addresses and check package versions. Slow or failed requests could break module and pipeline CI.

The proposed automation used the Wave API during development instead. Pipeline runs would not depend on that API because they would pull a pinned image directly.

### Hosting responsibility

nf-core relied on free public hosting from Quay.io and Galaxy. Service problems were outside the community's control.

Seqera Containers used infrastructure operated by Seqera and funded in part by AWS. The proposal gave nf-core a direct path for reporting build and hosting problems.

## How Seqera Containers differed

[Wave](https://seqera.io/wave/) builds a container when a package combination is requested. Seqera Containers adds long-term storage and a stable address for that result.

BioContainers builds when a Bioconda package is published. The proposed nf-core flow instead began when automation read a module's environment file. This made multi-package images and packages from other Conda channels easier to request.

At the time of the proposal, the services differed in these relevant ways:

- BioContainers built after Bioconda publication and used a mulled workflow for package combinations.
- Wave built direct requests on demand, produced lock files and build evidence, then kept images in a temporary cache.
- Seqera Containers stored those Wave builds behind stable addresses.
- BioContainers and Seqera Containers required only a container engine from pipeline users. Direct Wave use needed a Wave client.

## The proposed build flow

The architecture assigned one action to each participant:

1. A module developer edits `environment.yml`.
2. GitHub Actions asks Wave to solve the environment and build both image formats.
3. Seqera Containers stores the artifacts and build evidence.
4. nf-core records the image address and Conda lock file in module metadata.
5. Pipeline configuration pins those recorded values.
6. A user pulls the image with Docker or Singularity.

The image address identified a stored artifact. A later build could not silently replace the image already pinned by a pipeline release.

## Reproducibility and evidence

Three records served different purposes.

- The pinned image address selected the same stored container.
- The Conda lock file listed exact package files and hashes.
- The build page exposed inputs, logs, recipes, security scans, and a software bill of materials.

The lock file also let Conda users install an exact package set without solving the environment again. These controls improved software-environment repeatability. They did not, by themselves, guarantee reproducible scientific results.

An explicit Conda lock file looked like this:

```yaml
# platform: linux-64
@EXPLICIT
https://conda.anaconda.org/conda-forge/linux-64/_libgcc_mutex-0.1-conda_forge.tar.bz2#d7c89558ba0495403155b64376d81
https://conda.anaconda.org/conda-forge/linux-64/libgomp-14.1.0-h77fa898_1.conda#23c255b008c4f2ae008f81edcabaca89
# ... truncated ...
```

## Governance and timeline

nf-core maintainers and Seqera developers discussed the community's storage and build needs for more than a year. Seqera launched Seqera Containers in spring 2024. The nf-core steering group, core team, and maintainers then reviewed the migration proposal.

The remaining work covered module automation, pipeline configuration, bulk module updates, and pipeline updates. The team tracked that work in [nf-core/modules issue 5832](https://github.com/nf-core/modules/issues/5832).

At publication, the goal was to update modules by the end of 2024. That date and every future-tense statement in this post describe the proposal at that time.

## Limits and special cases

Wave is open source and can target other registries. That reduced dependence on one build implementation, but moving stored images would still require migration work and new addresses. We did not treat a registry change as a single risk-free configuration edit.

Some tools used custom Docker images instead of BioContainers. Those exceptions would continue. Organizations could also mirror pinned images into their own registry.

## What the proposal meant

### For pipeline users

Pipeline commands and container engines would remain the same. Conda users would receive exact lock files. Support for more CPU architectures was a goal, not a guarantee for every module at publication.

### For module developers

The environment file would become the main software definition. Automation would build images, save evidence, and propose version updates. Developers would spend less time creating mulled images or waiting for a second image format.

The proposal aimed to simplify container maintenance without hiding how an image was built. [Part 2](/posts/seqera-containers-part-2/) explains how nf-core planned to connect these pieces.
