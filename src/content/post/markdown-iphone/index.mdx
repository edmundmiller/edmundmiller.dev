I spent years waiting for the right combination of Org mode and a static site generator. Each experiment proved that I could build a custom publishing system, but it did not help me publish.

The same habit shaped my choice of phone. Running GrapheneOS was an interesting project, yet maintaining it made ordinary tasks less reliable. I eventually chose an iPhone and Apple Watch because their default features worked for the people around me.

## Choosing the boring path

I made the same choice for this site. Astro already supported the content and components I needed, so I started from an existing blog template and customized it.

MDX also gave me a practical boundary. I could write ordinary Markdown and add a component only when a post needed one. [Astro explains this content-first approach](https://docs.astro.build/en/concepts/why-astro/) without requiring one front-end framework for every page.

The tools did not need to prove anything about me. They only needed to keep publishing dependable.

## Keeping my editor

Choosing MDX did not require abandoning Emacs. Tailscale's guide to [configuring Emacs for MDX](https://tailscale.dev/blog/configuring-emacs-mdx) showed a small setup for editing MDX as Markdown. That was enough for prose, links, and fenced code.

Xe Iaso's account of [rebuilding a personal site](https://xeiaso.net/blog/xesite-v4/) also helped me separate the joy of making a site from the job of publishing words. Both can matter, but they do not have to happen at the same time.

I still enjoy unusual tools and custom systems. For this blog, however, Markdown was enough.
