## The architecture plan

In October 2024, nf-core planned to move module image builds from BioContainers to Wave and Seqera Containers. [Part 1](/posts/seqera-containers-part-1/) explains why. This post records the proposed implementation for maintainers and infrastructure contributors.

Ordinary pipeline use would change little. Module contributors would edit `environment.yml`, while automation handled the image records and pipeline configuration.

<Admonition type="tip" title="The October 2024 plan">

- Wave builds Docker and Singularity images for `linux/amd64` and `linux/arm64`.
- Each build produces a Conda lock file and build evidence.
- Module metadata records image addresses, build IDs, and scan IDs.
- nf-core/tools generates pipeline configuration from that metadata.
- nf-test checks whether package updates change module output.
- Renovate proposes safe package updates.

</Admonition>

Every future-tense statement below describes that plan. It is not a claim about the current rollout state.

## User-visible profiles

The existing `docker`, `podman`, `shifter`, and `charliecloud` profiles would keep using `linux/amd64` Docker images. New `_arm` variants would select `linux/arm64` images.

The `singularity` and `apptainer` profiles would use downloaded SIF images. New `_oras` variants would pull through the `oras://` protocol. Conda and Mamba profiles would use lock files, while `_env` variants kept direct `environment.yml` solving.

This profile split let users choose an engine, architecture, and transfer method without changing module code.

## Conda lock files

A Conda lock file lists every package file and hash in one solved environment. Conda can recreate that environment without solving its dependencies again.

```yaml title="FastQC Conda lock file for linux/amd64"
# platform: linux-64
@EXPLICIT
https://conda.anaconda.org/conda-forge/linux-64/_libgcc_mutex-0.1-conda_forge.tar.bz2#d7c89558ba0495403155b64376d81
https://conda.anaconda.org/conda-forge/linux-64/libgomp-13.2.0-h77fa898_7.conda#abf3fec87c2563697defa759dec3d639
# ... more exact package files ...
```

The proposed build stored separate lock files for each CPU architecture. Those files supported Conda users and made CI environments more stable.

## Singularity over HTTPS or ORAS

ORAS lets Singularity and Apptainer pull from an OCI registry. It moves each pull into the process task and supports registry authentication.

The 2024 plan kept HTTPS downloads as the default. Existing shared-cache variables worked with that path, and offline downloads did not require Singularity. Optional `_oras` profiles exposed the newer route for users who wanted it.

## Module records

The module remained the source of each process's software requirements. Automation connected three files.

### `environment.yml`

A contributor changed only the requested packages:

```yaml title="environment.yml"
channels:
  - conda-forge
  - bioconda
dependencies:
  - bioconda::fastqc=0.12.1
  - bioconda::samtools=1.16.1
```

### `main.nf`

Automation replaced conditional container logic with one generated default image:

```diff title="main.nf"
 process FASTQC {
     conda "${moduleDir}/environment.yml"
+    container "fastqc:0.12.1--5cfd0f3cb6760c42"
-    container "${ workflow.containerEngine == 'singularity' ?
-        'https://depot.galaxyproject.org/singularity/fastqc:0.12.1' :
-        'biocontainers/fastqc:0.12.1' }"
 }
```

Keeping the default address in `main.nf` made the requirement visible. It also gave nf-core/tools a key for matching the module to a pipeline process.

### `meta.yml`

Automation appended the full build records to module metadata:

```yaml title="meta.yml"
containers:
  docker:
    linux_amd64:
      name: community.wave.seqera.io/library/fastqc:0.12.1--5cfd0f3cb6760c42
      build_id: 5cfd0f3cb6760c42_1
      scan_id: 6fc310277b74
    linux_arm64:
      name: community.wave.seqera.io/library/fastqc:0.12.1--d3caca66b4f3d3b0
      build_id: d3caca66b4f3d3b0_1
  singularity:
    linux_amd64:
      name: oras://community.wave.seqera.io/library/fastqc:0.12.1--0827550dd72a3745
      https: https://community-cr-prod.seqera.io/example-fastqc.sif
      build_id: 0827550dd72a3745_1
  conda:
    linux_amd64:
      lock_file: https://wave.seqera.io/v1alpha1/builds/5cfd0f3cb6760c42_1/condalock
```

Build and scan IDs linked the image to logs, recipes, security scans, and its software bill of materials.

## Generated pipeline configuration

Nextflow does not read an nf-core module's `meta.yml`. nf-core/tools therefore needed to create one config file per engine and architecture.

```groovy title="config/containers_docker_amd64.config"
// AUTOGENERATED CONFIG BELOW THIS POINT - DO NOT EDIT
process {
    withName: 'NF_PIPELINE:FASTQC' {
        container = 'fastqc:0.12.1--5cfd0f3cb6760c42'
    }
}
```

The Conda config pointed each process to its lock file:

```groovy title="config/conda_lock_files_amd64.config"
// AUTOGENERATED CONFIG BELOW THIS POINT - DO NOT EDIT
process {
    withName: 'NF_PIPELINE:FASTQC' {
        conda = 'https://wave.seqera.io/v1alpha1/builds/5cfd0f3cb6760c42_1/condalock'
    }
}
```

`nextflow.config` included the matching file for the selected profile. Pipeline developers could place manual exceptions above the generated marker. Regeneration replaced only the section below it.

## Module automation

An `environment.yml` change started one sequence:

1. GitHub Actions detected the changed environment.
2. `wave-cli` requested images and lock files.
3. The workflow updated `main.nf` and `meta.yml`.
4. Stub tests recorded the reported software versions.
5. nf-test compared the new output with the previous snapshot.
6. Safe generated changes were committed back to the pull request.

![Container creation flow](https://raw.githubusercontent.com/nf-core/website/main/sites/main-site/src/content/blog/2024/images/seqera-containers-part-2/creation_flow.excalidraw.svg)

### The nf-test safety gate

nf-test snapshots record process output and software versions. The automation accepted an updated snapshot only when the version section changed. Any other output change left the test failing for a maintainer to review.

```json title="main.nf.test.snap"
{
  "versions": {
    "BOWTIE_ALIGN": {
      "bowtie": "1.3.0",
      "samtools": "1.16.1"
    }
  }
}
```

That gate let automation update expected versions without approving a changed scientific result.

### Renovate updates

Comments in `environment.yml` told Renovate how to find each Conda dependency:

```yaml
dependencies:
  # renovate: datasource=conda depName=bioconda/bwa
  - bioconda::bwa=0.7.18
  # renovate: datasource=conda depName=bioconda/samtools
  - bioconda::samtools=1.20
```

Renovate would open a version-bump pull request. The container workflow and nf-test checks would then run. Passing tests allowed automatic merge; failures requested the module's CODEOWNER.

![Container renovation flow](https://raw.githubusercontent.com/nf-core/website/main/sites/main-site/src/content/blog/2024/images/seqera-containers-part-2/renovate_flow.excalidraw.svg)

## Pipeline automation and exceptions

Installing, updating, or removing a module would regenerate the pipeline config files. The plan used `nextflow inspect` to map imported modules to process names.

At publication, `nextflow inspect` relied on a dry run and could miss skipped processes. Resolving that limitation was a prerequisite for the migration.

Some modules could not use Bioconda because of licenses, GPU variants, or custom builds. Local modules also lacked the shared automation. Pipeline maintainers would keep explicit container records for those cases.

Custom records could remain above the generated marker:

```groovy
// Manual container declarations stay above this line.
// AUTOGENERATED CONFIG BELOW THIS POINT - DO NOT EDIT
```

## Offline downloads

`nf-core download` needed to discover all images after profile configuration was applied. The plan reused the improved `nextflow inspect` output, then downloaded each resolved address as before.

This removed the need for the downloader to parse complex container expressions from module source.

## Roadmap recorded in October 2024

The original plan depended on several tracked changes:

- Nextflow: make `nextflow inspect` return every process.
- nf-core/tools: lint Renovate comments and version snapshots.
- nf-core/tools: generate pipeline container config files.
- nf-core/tools: rewrite [`nf-core download`](https://github.com/nf-core/tools/issues/3179).
- nf-core/modules: add [Renovate comments](https://github.com/nf-core/modules/issues/6504).
- nf-core/modules: add [Seqera Containers build automation](https://github.com/nf-core/modules/issues/6694).
- nf-core/modules: [bulk-update existing modules](https://github.com/nf-core/modules/issues/6698).

The end-of-2024 target belonged to this proposal. The linked issues remain the source for actual implementation status.
